# data compiance tool box
## 方案 A：Apple Editorial Workspace 结构稿说明

- 版本：v0.1
- 日期：2026-03-14
- 状态：结构方案确认稿
- 说明：本文件仅定义 UI 结构与视觉方向，不涉及代码实现

## 1. 方案定位

方案 A 的目标不是把当前工具做成营销页，而是把它重构成：

`具有 Apple 官网式高级感、但服务于真实填写任务的合规工作台`

核心原则：

- 视觉像 Apple
- 结构像工作台
- 操作像专业工具

## 2. 页面整体骨架

整页分为 5 个连续区块：

1. 顶部悬浮导航
2. 首屏任务 Hero
3. 产品/服务上下文区
4. 数据处理活动主舞台
5. ICO 预览区

页面滚动节奏：

- 首屏建立气质和主任务
- 中段承接记录上下文
- 主舞台完成核心填写
- 底部完成预览与校验

## 3. 顶部悬浮导航

### 3.1 布局

桌面端采用单行悬浮导航，宽度不满屏，居中悬浮。

从左到右：

- 产品名：`data compiance tool box`
- 角色切换：`Controller / Processor`
- 右侧工具区：`Trash / EN 中文 / Saved locally`

### 3.2 视觉

- 背景：半透明浅白玻璃层
- 边框：极轻灰线
- 圆角：全胶囊或 24px
- 阴影：轻微漫散阴影
- 高度：紧凑，不超过 64px

### 3.3 交互

- 页面下滚时导航固定吸附
- 当前角色高亮
- 语言切换与垃圾箱为次级控件
- 保存状态以小字显示，不抢注意力

## 4. 首屏任务 Hero

### 4.1 目标

首屏不再承担营销宣传，而承担三个任务：

- 告诉用户当前在做什么
- 告诉用户当前在处理哪个产品/服务
- 给用户一个明确的主操作入口

### 4.2 内容结构

左侧内容：

- 眉标题：`ICO-aligned privacy operations workspace`
- 主标题：
  - 中文示例：`记录你的数据处理活动`
  - 英文示例：`Document processing activities with clarity`
- 副标题：
  - 1 到 2 行，说明该工具基于 ICO ROPA 模板
- 当前上下文标签：
  - 当前角色
  - 当前产品/服务
  - 自动保存状态
- 按钮区：
  - 主按钮：`Add activity`
  - 次按钮：`Export JSON`

右侧内容：

- 一张精致的摘要卡片，不放大段文案
- 展示 3 到 4 个关键信息：
  - 当前记录名称
  - 处理活动数量
  - 最近保存时间
  - 当前模板类型

### 4.3 视觉要求

- 主标题应明显放大，形成 Apple 式视觉锚点
- 副标题控制在较短宽度内
- 按钮数量严格控制
- 右侧摘要卡片更像“产品状态面板”，不是普通数据卡片组

## 5. 产品/服务上下文区

### 5.1 定位

这是一个过渡区，用于承接首屏和主舞台。

它负责：

- 当前记录命名
- 新建产品/服务
- 历史记录切换

### 5.2 布局方案

桌面端建议改成左右双区：

- 左侧：当前记录编辑卡
- 右侧：历史记录横向或轻量纵向卡片列表

而不是现在这种标准侧栏式后台布局。

### 5.3 当前记录编辑卡

内容：

- 小标题：`Current product / service`
- 大号记录名称输入框
- 两个按钮：
  - `New record`
  - `Delete current`

设计要求：

- 输入框要更像 Hero 内的一部分
- 不是普通后台输入框
- 当前记录名称应有更高视觉权重

### 5.4 历史记录卡片

每张卡片包含：

- 记录名
- 活动数量
- 最近修改时间
- 操作入口：`Rename / Delete`

交互策略：

- 选中态清晰
- hover 时才强化操作按钮
- 默认保持安静，避免像文件管理器

## 6. 数据处理活动主舞台

### 6.1 定位

这是页面的主舞台，视觉上要占绝对中心。

需要做到：

- 看起来像高质量内容编辑区
- 不像老式系统表单
- 每条活动都自成一个完整模块

### 6.2 顶部工具条

工具条只保留必要内容：

- 左侧：`Processing Activities`
- 中部：当前记录小徽章
- 右侧：`Add activity`

可选：

- 当前筛选或排序状态

### 6.3 活动卡片结构

每条活动卡片结构建议如下：

1. 卡片头部
2. 卡片元信息
3. 分段内容区
4. 卡片底部操作区

#### 卡片头部

内容：

- 活动名称
- 活动编号
- 锁定状态

右侧操作：

- `Lock / Unlock`
- `Duplicate`
- `Delete`

视觉要求：

- 顶部区域整洁、留白大
- 不使用重边框
- 不使用过强色条

#### 卡片元信息

展示：

- 已完成字段数
- 当前 section 数量
- 自动保存提示

元信息应该是弱化信息，不可抢标题。

#### 分段内容区

按现有业务 sections 展开：

- Core record
- Transfers and security
- Lawful basis and rights
- Governance links

每个分段要像 Apple 页面中的“内容模块”，而不是传统 fieldset。

建议：

- 分段标题大于当前版本
- 分段说明减少字数
- 分段之间加大留白
- 每段内部字段卡片统一

### 6.4 字段设计策略

字段设计统一为“低噪音组件”：

- 输入框边界更轻
- 焦点态更干净
- 标签与帮助文案分层更清晰
- 帮助文案默认弱化

对于表格字段：

- 行高统一
- 表头更简洁
- 操作按钮减弱
- 整个容器像一张高质量白卡

### 6.5 锁定态设计

锁定状态不要只是“灰掉”。

推荐表达方式：

- 卡片头部出现简洁锁定徽章
- 字段仍保持可读，但交互禁用
- 背景轻微变化
- 操作按钮只保留 `Unlock`

## 7. ICO 预览区

### 7.1 定位

预览区是专业性最强的部分，必须保留，但要从“工具表格”升级为“审阅区”。

### 7.2 结构

顶部：

- 左侧：标题 `ICO-style record preview`
- 右侧：视图切换
  - `Current product / service`
  - `Global activities`

主体：

- 一张高质量表格卡片

### 7.3 视觉

- 表头背景淡灰
- 行高更大
- 列间距更舒适
- 粘性表头保留
- 当前产品/服务视图时减少不必要列噪音

### 7.4 体验

- 该区更像“最后校对区”
- 可读性优先于密度

## 8. 垃圾箱设计

### 8.1 位置

垃圾箱放在右上角，与语言切换同层。

### 8.2 表达方式

不建议一直展开成完整面板。

推荐两种方式中二选一：

- 方式 1：顶部小胶囊按钮，点击后下拉面板
- 方式 2：顶部轻量卡片，默认展示最近 1 到 2 个删除记录

我更推荐：

`方式 1：点击展开的下拉垃圾箱`

原因：

- 更接近 Apple 的顶层工具处理方式
- 更安静
- 不会抢占首屏右上角空间

### 8.3 内容

垃圾箱展开后每条记录展示：

- 记录名称
- 活动数量
- 删除时间
- `Restore`

## 9. 视觉规范建议

### 9.1 色彩

主方向：

- 页面背景：`#f5f5f7` 附近
- 主卡片背景：`#ffffff`
- 次级背景：极浅灰
- 文字主色：近黑
- 强调色：冷静蓝灰
- 警示色：低饱和橙

### 9.2 字体层级

- Hero 标题：48px 到 72px
- 区块标题：28px 到 40px
- 卡片标题：22px 到 28px
- 正文：15px 到 17px
- 标签：12px 到 13px

### 9.3 圆角

- 顶部导航：999px
- 大模块：28px
- 内容卡片：24px
- 字段：16px
- 表格单元：12px 到 14px

## 10. 动效建议

### 10.1 必要动效

- 首屏进入淡入
- 卡片 hover 微抬升
- 按钮 hover 轻微高亮
- 活动展开收起平滑
- 垃圾箱下拉淡入

### 10.2 限制

- 不做炫技动效
- 不做复杂视差
- 不做高频反馈

## 11. 响应式布局

### 11.1 桌面端

- 最强表现版本
- Hero 左右分栏
- 主舞台占最大空间

### 11.2 平板端

- Hero 上下排布
- 历史记录区缩短
- 顶部工具区仍保持同层

### 11.3 手机端

- 顶部导航折叠简化
- 垃圾箱改为按钮入口
- Hero 单列
- 所有活动卡片与字段转单列

## 12. 需要先确认的具体 UI 决策

在进入设计稿和代码之前，需要你确认以下 4 个点：

1. 顶部垃圾箱是否接受改成“点击展开下拉”，而不是一直固定展开
2. 首屏 Hero 是否接受从当前风格改成更强的任务引导式大标题布局
3. 历史记录区是否接受从左侧栏改成更轻量的横向/卡片化布局
4. 整体主色是否接受从当前暖棕色切换到更接近 Apple 的冷白灰体系
