# data compiance tool box
## Apple 风格 UI 改版产品需求文档

- 文档版本：v0.1
- 编写日期：2026-03-14
- 适用范围：`data compiance tool box` Web 端 UI 改版规划
- 当前状态：需求规划中，未进入代码修改阶段

## 1. 项目背景

当前 `data compiance tool box` 已具备基础业务能力：

- 支持 `Controller / Processor` 模式切换
- 支持产品/服务记录管理
- 支持数据处理活动填写、锁定、自动保存、JSON 导出
- 支持中英文切换

现阶段主要问题不在业务功能，而在界面表现：

- 页面信息密度较高，缺少明确的视觉主次
- 顶部、表单区、历史记录区之间的节奏不够统一
- 虽然功能完整，但产品感不够强，缺少高端、克制、可信赖的视觉体验
- 现有界面更像“表单集合”，而不是“专业合规工作台”

本次改版目标是参考 [Apple 中国官网](https://www.apple.com.cn) 的视觉语言和页面组织方式，对现有工具进行一轮系统性的 UI 重构。

## 2. 参考网站研究结论

### 2.1 参考来源

- 访问站点：[Apple 中国官网](https://www.apple.com.cn)
- 本次提取的核心结构特征来自首页与全局导航模式

### 2.2 抽取到的视觉特征

Apple 中国官网首页的主要特征：

- 顶部导航极简，横向展开，弱化边框，强调留白
- 首屏使用大标题、短副标题、极少量按钮，信息控制非常克制
- 页面采用“连续全宽分区”结构，而不是一个巨大的后台表单
- 每个模块只表达一个主题，文案短，聚焦度高
- 第二屏以后大量使用双列卡片区块，形成节奏变化
- 大面积浅色背景搭配少量高对比深色区块，形成视觉呼吸感
- CTA 常以两个并列动作出现，例如“进一步了解 / 购买”
- 字体层级清晰，大标题非常大，正文非常短
- 圆角、阴影、分隔都非常轻，避免传统后台产品那种“重边框感”
- 交互动效克制，只在 hover、滚动显现、模块进入时体现轻微反馈

### 2.3 首页内容结构特征

从页面内容和源码可见的首页模块特征包括：

- 顶部全局导航：Apple、商店、Mac、iPad、iPhone、Watch、Vision、AirPods、家居、娱乐、配件、技术支持
- 首页主模块以单一产品 hero 为核心，例如：
  - `MacBook Neo`
  - `iPhone 17e`
  - `MacBook Pro`
  - `MacBook Air`
  - `iPad Air`
  - `Studio Display XDR`
  - `Studio Display`
- 页尾信息大量折叠，正文区保持高度简洁

### 2.4 可借鉴的不是“像苹果”，而是这些方法

- 用更少的元素表达更强的层级
- 用模块化叙事代替一次性把所有信息摊开
- 用高质量留白提升可信度
- 用一致的组件节奏建立产品完成度

## 3. 设计目标

本次 UI 改版目标：

1. 将当前工具改造成“高端、安静、专业”的合规工作台
2. 保留现有信息架构和业务流程，不先动核心业务逻辑
3. 让用户在视觉上先看到“当前任务”，再看到“支撑信息”
4. 明显降低后台表单工具的压迫感
5. 让中英文切换、垃圾箱、产品/服务切换都更自然融入顶层布局
6. 让页面在桌面端与移动端都具备完整的观感统一性

## 4. 非目标

本轮不做的事：

- 不重写现有业务逻辑
- 不先改导出格式
- 不先接入数据库
- 不先做账号体系
- 不先做多人协同
- 不直接复制 Apple 的图片、图标、营销文案或品牌资产

说明：
本项目借鉴的是 Apple 网站的布局语言、节奏和视觉方法，不是 1:1 克隆品牌资产。

## 5. 产品定位

改版后的产品定位应从“合规表单页”升级为：

`为隐私法务、数据合规、产品和安全团队服务的高质感 ROPA 工作台`

核心感知关键词：

- Calm
- Premium
- Trustworthy
- Precise
- Minimal
- Structured

## 6. 核心用户

### 6.1 主用户

- 企业隐私合规负责人
- DPO / Data Protection Team
- 法务与安全合规人员
- 产品经理与运营负责人

### 6.2 使用场景

- 按产品/服务录入 ROPA
- 进行内部合规梳理
- 为 DPIA、审计、对外问询做准备
- 定期维护和更新数据处理活动记录

## 7. 总体设计策略

### 7.1 页面策略

将当前页面改造成三层结构：

1. 顶部全局层
2. 当前工作焦点层
3. 支撑信息与记录层

### 7.2 视觉策略

- 顶部区域采用更简洁、近似 Apple 的悬浮导航风格
- 首屏不再只是一个普通 hero，而是一个“任务引导区”
- 表单区不再像后台系统块状堆叠，而要变成“连续分段、轻背景、高留白”的工作流
- 预览区需要像 Apple 卡片一样更整洁、更克制，而非传统表格工具感

### 7.3 交互策略

- 强化“当前产品/服务”的上下文
- 历史记录和垃圾箱弱化为辅助操作，不抢主体视觉
- 当前处理活动卡片必须更像高质量内容模块，而不是展开/折叠表单容器
- 自动保存状态要更轻、更隐形，但要始终可感知

## 8. 页面信息架构建议

## 8.1 顶部全局区

应包含：

- 产品名称：`data compiance tool box`
- 角色切换：`Controller / Processor`
- 语言切换：`EN / 中文`
- 垃圾箱入口
- 轻量状态提示，例如 `Saved locally`

设计要求：

- 顶部导航高度收敛
- 模糊玻璃或半透明浅色背景可保留，但需更简洁
- 垃圾箱与语言切换同层级放在右侧
- 顶部区必须支持页面滚动后的固定吸附

## 8.2 首屏任务引导区

该区域取代当前偏营销式 hero，改为“任务型 hero”。

建议包含：

- 大标题：说明当前页面在做什么
- 一行副标题：强调基于 ICO 模板记录 ROPA
- 当前产品/服务名称
- 主动作按钮：`Add activity`
- 次动作按钮：`Load sample` 或 `Export JSON`
- 当前角色说明

设计要求：

- 视觉像 Apple 首屏 hero，标题大，正文短
- 不出现过多辅助说明文字
- 主动作按钮数量控制在 1 到 2 个
- 桌面端保持宽松留白，移动端转为垂直堆叠

## 8.3 产品/服务记录区

该区域是支撑导航区，不是主舞台。

需要保留：

- 当前产品/服务名称编辑
- 新建记录
- 历史记录列表
- 每条记录的重命名、删除

设计要求：

- 历史记录应改成更轻量的卡片列表
- 当前选中项有明显但克制的高亮
- 操作按钮只在 hover 或聚焦时更明显
- 记录名称过长时应优雅截断

## 8.4 数据处理活动主区

这是整个产品的核心区域。

设计要求：

- 采用 Apple 风格的连续模块布局
- 每个活动卡片像一个完整 section
- 卡片间距要足够大
- 每条活动的标题区域非常清楚
- 锁定状态以简洁图标和状态文字表达
- 展开/收起的交互必须更平滑

建议每条活动结构：

1. 标题区
2. 状态元信息区
3. 分段表单区
4. 底部操作区

## 8.5 预览区

当前 ICO 风格记录表是功能必需，但视觉可重做。

设计要求：

- 不取消表格
- 将表格容器做成更像 Apple 信息卡的高质量容器
- 表头与行高统一
- 视图切换按钮放右上角，但样式需更轻
- 当前产品/服务与全局视图切换要更明显

## 9. 视觉设计规范

### 9.1 设计气质

整体气质要求：

- 明亮
- 克制
- 高级
- 低噪音
- 专业可信

避免：

- 过重描边
- 过度阴影
- 大量渐变堆叠
- 后台模板化的网格块感
- SaaS 通用紫色风格

### 9.2 色彩方向

建议采用浅色体系，基于中性色和暖灰色构建：

- 页面背景：近白暖灰
- 主卡片：纯白或极浅米白
- 文字主色：深灰黑
- 次级文字：中灰
- 强调色：克制的深蓝灰或石墨黑，不建议用高饱和橙红作为主视觉
- 成功状态：柔和绿色
- 警示状态：柔和琥珀色

建议不要继续沿用当前偏暖棕色为主的整站主色方案。

### 9.3 字体策略

建议模仿 Apple 的层级方式：

- 大标题：高辨识、高留白、极简
- 正文：短句、窄宽度、舒适行高
- 标签：小字号、弱化
- 数据和状态：中等字重，避免过粗

字体方向：

- 中文：优先系统中文字体栈
- 英文：优先 SF Pro 风格近似栈或高质量系统 sans-serif
- 标题不建议继续使用过强衬线字体

### 9.4 圆角与边框

- 外层模块圆角：20px 到 28px
- 内层卡片圆角：16px 到 22px
- 输入框圆角：14px 到 18px
- 边框非常轻，更多依靠分层与背景对比建立结构

### 9.5 阴影

- 阴影应轻、散、低对比
- 不允许出现厚重悬浮卡片感
- 阴影主要用于区分层级，不用于制造戏剧化效果

## 10. 布局规范

### 10.1 桌面端

- 页面最大宽度建议：`1440px - 1600px`
- 内容主体区采用单列主舞台 + 轻量侧区
- 首屏 hero 与主任务区之间要有较大垂直留白

### 10.2 平板端

- 顶部导航折叠但不失去层级
- 侧栏与主区可转为上下布局
- 活动卡片仍应保留完整分段

### 10.3 移动端

- 首屏转为单列
- 历史记录区和垃圾箱默认弱化
- 表单字段保持单列
- 核心 CTA 保持可见

## 11. 组件改版需求

### 11.1 顶部导航

需求：

- 改成更轻、更平、更克制的胶囊式导航
- 滚动时保持吸附
- 各控件间距统一

### 11.2 按钮

需求：

- 主按钮使用深色实体按钮
- 次按钮使用弱背景或文字按钮
- 所有按钮圆角统一
- 不同按钮优先级必须明显

### 11.3 输入框

需求：

- 高度统一
- 文本输入、下拉框、URL 输入统一视觉
- 焦点态应简洁、清晰，不要使用大面积彩色光晕

### 11.4 活动卡片

需求：

- 当前色块区分保留，但要更克制
- 不使用过强左边彩条
- 可改为顶部细色带、状态徽章或轻背景分区
- 锁定状态要更明确，但不能让界面显脏

### 11.5 表格

需求：

- 接收方表格与跨境传输表格视觉统一
- 表头固定风格
- 单元格高度统一
- 行内按钮更轻量

### 11.6 Tooltip

需求：

- 感叹号 tooltip 风格统一
- 显示内容简短、专业、易读
- Hover 与 focus 都可触发

## 12. 动效需求

参考 Apple 风格，动效应存在，但非常克制。

建议加入：

- 首屏模块淡入上浮
- 活动卡片进入时轻微渐显
- hover 时按钮和卡片只做 1 到 2px 的微弱位移或阴影变化
- 详情展开收起时有平滑过渡

禁止：

- 大范围弹跳
- 高频闪动
- 明显“炫技型”动效

## 13. 文案策略

页面文案应更短、更准、更接近产品级表达。

要求：

- 标题短句化
- 辅助文案控制在一到两行
- 字段说明能省则省
- 只在确实会影响理解时才展示帮助文案

中英文要求：

- 同一层级文案长度需控制
- 避免中文很长、英文很短导致布局不平衡

## 14. 可访问性要求

- 中英文切换对屏幕阅读器友好
- Tooltip 支持键盘 focus
- 对比度满足基础可访问性要求
- 所有按钮与输入具有清晰 focus 状态
- 不仅依赖颜色表达锁定、删除、当前选中等状态

## 15. 业务逻辑与 UI 的边界

本轮是 UI 设计驱动改版，不主动更改以下核心逻辑：

- Controller / Processor 数据模型
- 自动保存机制
- 锁定/解锁逻辑
- 记录删除/恢复逻辑
- JSON 导出逻辑
- 权利自动判断逻辑

允许为配合 UI 做的轻量调整：

- 组件结构重组
- 按钮层级和位置优化
- 顶部导航与垃圾箱重排
- 首屏信息重排
- 历史记录的展示方式优化

## 16. 产出物要求

进入设计和开发阶段后，应至少产出：

1. 高保真首页改版方案
2. 数据处理活动主区改版方案
3. 预览区改版方案
4. 桌面端与移动端响应式方案
5. UI token 方案

## 17. 验收标准

当以下条件满足时，可认为本轮 UI 改版成功：

1. 用户打开页面，第一眼感知为“高端工具”而不是“普通后台表单”
2. 用户能在 3 秒内识别当前角色、当前产品/服务、当前主任务
3. 页面层级明显优于当前版本
4. 历史记录和垃圾箱不再挤占主舞台
5. 数据处理活动卡片更易读、更统一
6. 预览区更清晰，但不削弱其专业性
7. 中英文切换后，页面布局仍然稳定
8. 桌面端和移动端都能保持完整的美感

## 18. 建议实施顺序

建议按以下顺序推进，而不是一次性重写：

### 阶段一：信息架构与框架重排

- 顶部导航重构
- 垃圾箱迁移至顶部
- 首屏 hero 重构
- 侧栏与主区层级重排

### 阶段二：核心组件重绘

- 按钮系统
- 表单系统
- 活动卡片
- 历史记录卡片
- 表格系统

### 阶段三：细节与动态表现

- 动效
- tooltip
- hover/focus
- 响应式

## 19. 待确认需求

在正式进入 UI 开发前，需要你确认以下事项：

1. 是否接受“参考 Apple 的结构与气质，但不复制品牌资产”的原则
2. 是否同意把当前首页 hero 改成更任务导向的首屏
3. 是否接受弱化左侧栏，让主舞台集中到数据处理活动区
4. 是否接受将整站主色从当前暖棕方向改为更中性、近 Apple 的浅色体系
5. 是否需要同时产出一版更强展示感的概念稿

## 20. 下一步建议

在你确认本 PRD 后，下一步应按以下顺序执行：

1. 先生成 2 到 3 个 Apple 风格 UI 方向稿
2. 从中选定一个视觉方向
3. 再进入 `index.html / styles.css / app.js` 的实际改版

## 22. 结论

本次改版不应理解为“把现有页面简单换皮”，而应理解为：

`借鉴 Apple 的信息组织、版式节奏和高级感表达，重做一个真正像产品的合规工作台。`

在该原则下，后续 UI 修改才会有统一标准。
